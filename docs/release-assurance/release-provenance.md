# Release provenance

ResearchOps uses a dedicated release provenance workflow for tagged releases.

The workflow generates local provenance evidence and requests a GitHub artifact attestation for the release provenance package.

## Workflow

`.github/workflows/release-provenance.yml`

The workflow runs on:

- tags matching `v*`
- manual dispatch

## Generated evidence

The workflow generates:

- `release-provenance-manifest.json`
- `slsa-provenance.json`
- `dsse-envelope.json`
- `researchops-release-provenance.tgz`

The generated DSSE-shaped envelope is not a trusted signature by itself. It is local evidence metadata.

Trusted provenance depends on the GitHub artifact attestation generated by the workflow.

## Required checks before release provenance

Before treating a tagged release as release-ready, confirm:

- Release Gate passed on the release commit.
- Accessibility audit passed on the release commit.
- Security audit policy has no blocking findings.
- Deployment toolchain evidence is current.
- Branch protection has been verified from live GitHub settings.
- Release evidence commit references match the release commit.

## Verification

After the workflow completes, verify the attestation with GitHub CLI:

```bash
gh attestation verify artifacts/researchops-release-provenance.tgz --repo kevinrapley/ResearchOps
```

If the artifact has been downloaded from GitHub Actions, run verification from the directory containing the artifact.

## Limitations

Artifact attestation availability depends on GitHub repository visibility and plan support.

Do not treat local provenance files as equivalent to a signed attestation. They are supporting evidence only.
