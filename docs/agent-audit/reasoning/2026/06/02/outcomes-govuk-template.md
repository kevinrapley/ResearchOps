## Findings

The current outcomes page was mixed-mode. It loaded GOV.UK Frontend, legacy GOV.UK CSS partials and `/css/screen.css` at the same time.

The page hand-coded breadcrumbs, form groups, inputs, selects, textarea, table and buttons. It did not have a Nunjucks source template in the GOV.UK page renderer.

The impact tracker script used `alert("Metric name is required")` for required field validation. That does not match a GOV.UK Frontend form error pattern.

Codex correctly identified that the rendered outcomes page must be committed from the template change and that route-state markers need to remain aligned with the breadcrumb/back-link test contract.

## Decisions

1. Add `src/govuk/templates/pages/projects-outcomes.njk` as the source template for `/pages/projects/outcomes/`.
2. Use GOV.UK macros for breadcrumbs, buttons, error summary, input, inset text, select, table and textarea.
3. Keep the existing IDs used by page scripts, including `impact-tracker`, `impact-form`, `impact-table`, `impact-metricName` and `impact-insightId`.
4. Register the new template in `scripts/govuk/render-govuk-pages.mjs`.
5. Do not hand-edit `public/pages/projects/outcomes/index.html`. Let the render workflow produce it from the template.
6. Keep the current breadcrumb/page marker as `Impact & ROI`.
7. Remove the action-bar `Back to Project` button from the outcomes template because the user explicitly rejected that UI.
8. Render the impact table through the `govukTable` macro rather than hand-coded table markup.
9. Replace browser alert validation in `impact-tracker.js` with a GOV.UK error summary path.
10. Update route-state expectations so outcomes is treated as a GOV.UK Frontend-rendered route, not a legacy `govuk-forms.css` route.
