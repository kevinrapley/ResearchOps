Modes

Mode modules are task runbooks. They define the correct workflow before an agent edits, reviews or releases repository work.

This page is generated documentation. The canonical source is .agent-operating-model/bundles/github/modes/.

GitHub bundleModes

Every mode has the same control shape: inputs, actions, outputs, failure states, required contracts and required graders.

Discovery

repo-discovery.xml builds verified understanding before changes. It requires root docs, manifests, workflows and governance to be read before implementation.

Instantiation

repo-instantiate.xml creates a governed repository foundation using the template registry, GitHub settings, conformance matrix, gap register and CI selection.

Build

repo-build.xml implements small reviewable changes with tests, docs, fixtures, evidence, conformance updates and PR summary.

Fix

repo-fix.xml diagnoses defects, fixes root causes, adds regression tests and records non-obvious traps in RECENT_LEARNINGS.md.

Review

repo-review.xml reviews PRs, CI evidence, inline comments and Codex comments. It blocks approval without evidence.

Security

repo-security.xml handles vulnerabilities, dependency review, CodeQL, secret scanning and workflow hardening.

Release

repo-release.xml validates version, changelog, release gates, SBOM, provenance, performance budgets and rollback.

Conformance

repo-conformance.xml audits repository files, workflows and settings against bundle controls.

Documentation

repo-docs.xml treats documentation as a tested surface with provenance and validation evidence.

Archive

repo-archive.xml supports safe deprecation, consumer assessment, migration notes and settings updates.

Mode source pattern

The source is intentionally regular so the agent can treat it as a checklist.

<mode id="repo-build" version="2.9.2">
  <title>Repository Build Mode</title>
  <purpose>Implement small reviewable changes.</purpose>
  <inputs>
    <item>Issue/task</item>
    <item>Discovery summary</item>
  </inputs>
  <actions>
    <item>Plan with developer control contract.</item>
    <item>Implement code, tests, docs and fixtures.</item>
    <item>Validate contracts and quality gates.</item>
    <item>Update conformance, gaps and recent learnings.</item>
  </actions>
  <outputs>
    <item>Diff</item>
    <item>Tests</item>
    <item>Evidence</item>
    <item>PR summary</item>
  </outputs>
  <failure_states>
    <item>No tests</item>
    <item>Docs stale</item>
    <item>Fabricated evidence</item>
  </failure_states>
  <required_contracts>
    <item>pull-request-contract.schema.json</item>
    <item>agent-evidence.schema.json</item>
  </required_contracts>
  <required_graders>
    <item>code-confidence-grader.xml</item>
    <item>pr-readiness-grader.xml</item>
  </required_graders>
</mode>

How the agent uses this

The mode tells the agent what counts as complete. For build work, a code diff alone is insufficient. Tests, evidence, PR summary and governance updates are also part of the mode.

Failure states

Failure states are blocking conditions. The agent must not call work complete if one remains true.
