1. Purpose & what changed in v2

The goal is not a better chatbot. It is to make ResearchOps observable to an AI agent, then let that agent build experiential knowledge from real product events, ground its rewards in observed environment deltas, plan over a small world model, and act inside the platform behind explicit approval gates.

v2 in one paragraph: v1 was a solid sketch but read as a generic Cloudflare tutorial. v2 is grounded in the actual ResearchOps repo (correct migration numbering, real binding names, real auth helper, real worker entrypoint), reuses what already exists (Workers AI binding, ai-rewrite.js pattern, Cucumber + Playwright), and treats the four Era of Experience principles — streams, grounded rewards, real actions, planning — as load-bearing rather than decorative.

Observe

The platform emits structured events when studies, guides, consent forms, sessions, notes, synthesis artefacts and recommendations change. Streams, not episodes.

Contribute

The agent drafts reviewable, evidence-linked recommendations for safeguarding, traceability, research questions, sampling, accessibility, notes and analysis gaps.

Learn

Two reward channels — environment-grounded and human-grounded — update action values. A planner predicts the next gap; outcomes refine the planner.

Operating principle: no fake outcomes, no simulated rewards, no agent value without platform evidence. The AI learns from observed ResearchOps activity, not from a toy loop. Every component below must be justifiable against one of the four principles in §2.

2. The four-principle backbone new in v2

Every component below must advance one of these. If it does not, it is out of scope.

PrincipleWhat it demands of ResearchOpsLives in
Streams, not episodesA continuous, lifelong experience trace per study/project — not isolated chat turns or single API calls.§6 D1 stream, §8 diary
Grounded rewardsReward comes from observed environment deltas (a safeguarding section did appear after the agent flagged it), not only from a human thumbs-up.§7 signals + credit assignment
Actions in the environmentThe agent drafts real artefacts (gap reviews, plans), behind approvals. It does not just produce conversational replies.§10 approvals, §18 actions
Planning over a world modelThe agent predicts where the next gap will appear and pre-positions effort. Outcomes verify or refute the prediction.§9 planning loop
v1 vs v2: v1 implemented the first three lightly and skipped the fourth. v2 treats all four as load-bearing and adds streams (stream_key, stream_seq, event_hash), explicit grounded reward (credit.js), and a planner that writes to rops_experience_predictions.

  • Rules-based recommendations with optional Workers AI reflection.
  • Feedback-driven and environment-driven action-value updates.
  • Approval-gated draft contributions (L0–L2 matrix).
  • Visible agent panel and experience cockpit (WCAG 2.2 AA).

Out of scope — hard boundaries

  • No autonomous participant contact.
  • No consent-wording changes.
  • No report publishing without human review.
  • No deletion of research records.
  • No model fine-tuning or weight updates.
  • No compliance attestations by the agent.
  • No writes to Airtable participant tables.

5. Architecture

The Experience Layer sits alongside the existing ResearchOps Worker and service modules. It uses the current Worker route pattern, adds an experience service module, persists events in D1, optionally stores semantic memory in Vectorize, runs a scheduled cron for the diary/evaluator, and returns recommendations to the UI.

LayerResponsibilityMVP implementation
ObservationMake platform activity visible. Idempotent.experience-client.js, observeExperience(), event_hash dedupe.
Experience streamCanonical record of events and consequences, ordered per stream.rops_experience_events with stream_key + stream_seq.
InferenceTurn observed gaps into useful interventions; predict the next one.Rules-first; planner predicts; Workers AI reflects.
LearningAdapt from two reward channels.credit.js attributes environment deltas; rewardFromFeedback attributes humans.
OversightPrevent unsafe autonomy via a transparent matrix.L0–L2 approval matrix in actions.js + approvals queue.
ReflectionNon-human reasoning, auditable in language.Daily cron writes rops_experience_diary.
ValidationProve the agent helps; defer rollout if it does not.Shadow mode + evaluator + replay scripts.

7. Grounded reward — the two channels new

v1's reward model conflated environment signal with human approval. v2 separates them explicitly.

Channel 1: environment-grounded

Reward derived from observed feature deltas (signals.js → rewardFromSignals). Did the world actually improve? Did hasSafeguardingSection flip false → true after a SAFEGUARDING_GAP was opened?

Channel 2: human-grounded

Reward derived from researcher feedback (rewardFromFeedback). Did a researcher accept, resolve, ignore, reject or mark harmful?

A new credit.js closes the loop: when a flip-event arrives within N hours of a related open recommendation, the environment delta is attributed back to the prior action and a synthetic feedback row is written with feedbackType = 'environment_resolved'. Feedback alone would be sycophancy bait; environment-grounded reward is what the paper calls for.

8. Agent diary — non-human reasoning new

A scheduled Worker (cron 0 2 * * *) wakes the agent and asks it to write a paragraph per active stream. Stored in rops_experience_diary and surfaced in the cockpit. This is the closest the MVP gets to the paper's "non-human reasoning" idea, because the diary is generated from the action-value table and predictions the agent maintains itself — not from researcher prose.

Example diary entry: “In the last 24 hours on study recXXX, I observed 7 events. I opened 3 recommendations — 2 traceability, 1 safeguarding. Researchers accepted the safeguarding draft and ignored one traceability. My estimate of draft_traceability_gap_report value for consent_form context dropped from 0.41 to 0.36; I will deprioritise it there. I predict a session_notes_gap after the next session.completed because the team has no notes uploaded yet this week.”
Prompt-injection defense: the diary prompt only ever ingests D1 aggregates and feature flags, never raw researcher prose or participant text. There is no path for participant content to influence the LLM call.

11. Privacy & DPIA new

v1's "don't store raw notes" callout was correct but vague. v2 makes it concrete and enforceable.

  • Allow-list, not block-list. The sanitiser accepts only known feature keys (booleans, counts, enums). Anything else is dropped before persistence. A unit test throws if an unknown key reaches storage.
  • No free text in before_json, after_json, derived_features_json. Free text is hashed to a category if needed.
  • Stable derived features only. Names, emails, postcodes, dates of birth, free-text quotes are never derived features.
  • DSAR endpoint. GET /api/experience/dsar?actorId=… (admin) returns every event/recommendation/feedback row where the user appears as actor_id; DELETE tombstones them.
  • Retention. Events 90 d · Recommendations 1 y · Feedback 1 y · Action values indefinite (no PII) · Diary 2 y · Predictions 30 d.
  • DPIA template. docs/dpia/experience-layer.md covers purposes, legal basis (legitimate interest, opt-out), categories of data, retention, risks, mitigations, sign-off.

14. Evaluation harness new

v1 had success measures but no way to compute them. v2 adds explicit scripts.

  • scripts/experience/evaluate.mjs — reads the last 30 days of events and feedback, computes per-recommendation-type precision/recall against accepted/resolved, prints a markdown report. Wired into npm run experience:eval.
  • scripts/experience/replay.mjs — replays a study's stream against the current rules/planner and reports differences vs. shipped recommendations. Used before any rules/planner change.
  • Cron runs the evaluator weekly and writes a summary diary entry with scope_key='global'.
Why this matters: it is what makes the agent's learning verifiable rather than vibes-based. Without it, the team has no way to tell whether action values are improving or drifting.

15. Shadow mode A/B new

For the first two weeks of Alpha 2, a random 30% of new recommendations are written with visibility='shadow'. They never render. After two weeks, compare gap-closure rate between studies that saw recommendations and those that did not.

If lift is statistically meaningful

Flip everyone to visibility='visible' and proceed to Alpha 3.

If not

The team has a real datum to redesign against, rather than a "the agent feels useful" claim.

Ethics: withholding potentially useful recommendations from 30% of studies for two weeks needs DPO sign-off. Fallback: per-recommendation A/B (30% of candidates shadowed) instead of whole studies.

Grounded reward attribution

Closes the loop between an open recommendation and the environment delta that resolves it.

review 
// infra/cloudflare/src/service/experience/credit.js
export async function attributeEnvironmentReward(env, event) {
    const openRecs = await listOpenRecommendationsForStream(env, event.streamKey);
    for (const rec of openRecs) {
        if (recommendationResolvedBy(rec, event)) {
            await insertFeedback(env, {
                id: createId("fb"),
                agentId: rec.agentId,
                recommendationId: rec.id,
                feedbackType: "environment_resolved",
                reward: 0.5,
                actorId: null,
                note: `auto: env delta ${describeDelta(rec, event)}`,
                createdAt: nowIso()
            });
            await updateActionValue(env, rec.agentId,
                contextKeyFromRecommendation(rec),
                rec.action?.name, 0.5);
        }
    }
}

Planner V0 (heuristic)

State-machine over recent events + action values. AI swap-in is V1.

review 
// infra/cloudflare/src/service/experience/planner.js
export function predictNextGaps(stream, actionValues) {
    const predictions = [];
    const latest = stream[stream.length - 1];
    const features = mergeFeatures(latest);

    if (features.hasVulnerableUsersFlag && !features.hasDistressProtocol &&
        hoursOpen(latest) > 48) {
        predictions.push({
            predictedEventType: "session.started",
            predictedRecommendationType: "safeguarding_gap",
            confidence: 0.72,
            rationale: "vuln flag set, no distress protocol, > 48h open"
        });
    }
    return predictions;
}

20. Risks & open questions

Cron + Workers AI cost

Daily diary across many active studies could exceed the Workers AI free tier. Rate-limit to max(active_streams, 50) per cron and back off if the previous run hit the budget guard. Decision needed: cost ceiling per day.

Airtable ↔ D1 drift

If features are derived from Airtable but the record changes underneath, the agent's view goes stale. Mitigation: include Airtable lastModifiedTime in event payloads.

claude/ branch policy conflict

This plan was authored on claude/… per session instruction, but AGENTS.md forbids that prefix for repo-affecting work. Confirmed: that branch carries only this plan file; all implementation moves to feature/experience-layer-mvp.

Shadow-mode ethics

Withholding potentially useful recommendations from 30% of studies for two weeks needs DPO sign-off. Fallback: per-recommendation A/B (30% of candidates) instead of whole studies.
